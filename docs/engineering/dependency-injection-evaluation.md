# 依赖注入方案评估：Koin vs Hilt（2026-03-26）

> 本文对应 Issue #60 的 `[ARCH-1]`：评估引入 Hilt/Koin 简化依赖注入。
>
> 当前工程已使用 Koin（例如 `koinViewModel`），因此这里的评估重点是：
> - 是否需要迁移到 Hilt（或引入其它框架）
> - 以及在什么条件下值得承担迁移成本

---

## 1. 当前状态（事实）

- UI 层通过 `koinViewModel()` 获取 ViewModel（Compose 侧）。
- 数据层与仓储层在 `AppContainer` 之外，整体已围绕“构造注入 + 门面仓储”拆分出较清晰边界。
- 同步、备份、提醒等高风险能力的测试多数采用 Fake/Recording 实现替换依赖，现有模式可用且覆盖面在增长。

结论：当前注入体系**能支撑开发与测试**，痛点主要是“可维护性/一致性”而不是“完全无法扩展”。

---

## 2. 评估维度

为了避免“迁移框架只是换一种写法”，评估以以下维度为准：

1. **开发体验与心智负担**：新页面/新用例接入成本、模块边界是否清晰。
2. **构建与增量编译**：注解处理/代码生成对构建时间的影响。
3. **错误暴露时机**：运行时注入失败 vs 编译期图校验。
4. **测试替换成本**：替换依赖是否自然，是否需要复杂的测试组件。
5. **迁移风险**：需要修改的文件数量与跨层耦合面。

---

## 3. Koin（现状）优劣

优点：

- 低侵入：无需大规模注解，适合当前“单模块 app + 清晰分层”结构。
- 测试替换直观：Fake 仓储/服务可以在测试里直接注入或通过构造注入替换。
- 迁移成本为零：现有代码与文档一致，继续优化（比如 reducer/delegate）不依赖 DI 框架变化。

缺点：

- 运行时失败：依赖图问题通常在运行时才暴露，需要靠测试与启动门禁兜底。
- 约束弱：团队规模扩大后更容易出现“模块里塞了不该塞的依赖”。

---

## 4. Hilt（候选）优劣

优点：

- 编译期图校验：依赖缺失更早暴露，尤其适合依赖关系复杂、多 module 的工程。
- 与 Android 组件集成更“标准”：部分团队更容易招人/对齐工程习惯。

缺点：

- 注解处理与生成代码：对构建速度与排错路径有额外成本。
- 迁移涉及面大：需要重写 ViewModel 注入、模块声明、测试组件/替换策略。
- 当前工程收益有限：目前依赖图复杂度尚未达到必须引入编译期框架的程度。

---

## 5. 结论与决策

结论（本阶段）：

- **继续使用 Koin**，不引入 Hilt/Koin 的迁移或双栈并存。
- 现阶段更高收益的优化方向是：统一状态更新模式（Reducer/Delegate）、减少样板与提升测试门禁，而不是替换 DI 框架。

触发重新评估的条件（建议写入工程门禁）：

- 工程拆成多个 Gradle module，且依赖图明显膨胀（跨层依赖开始难以追踪）。
- Koin 运行时注入错误在 CI/真机回归中出现频率上升，且难以通过测试策略覆盖。
- 需要更强的编译期约束来控制“领域层/数据层/feature 层”依赖边界。

---

## 6. 下一步建议（不迁移也能改进）

- 在 Koin Module 的定义处增加“分层依赖清单”（只列公开入口），减少隐式耦合。
- 为高风险流程（同步/备份/提醒）保持集成测试与 UI 内容测试，降低运行时注入问题被遗漏的概率。
