# 过期任务衰减机制设计

## 1. 文档目的

本文件用于重新定义忆刻在“长时间未复习”场景下的处理原则。

这里不再使用“惩罚用户”作为设计出发点，而改为描述为：

- 卡片在长时间过期后，其**记忆可信度下降**
- 系统需要通过**阶段衰减**帮助用户重新建立记忆
- 过期处理的目标是**疏导积压**，而不是制造挫败

---

## 2. 当前调度规则回顾

### 2.1 现有阶段数组

忆刻当前采用固定阶段数组：

```text
[1, 2, 4, 7, 15, 30, 90, 180]
```

对应：

- stage 0 -> 1 天
- stage 1 -> 2 天
- stage 2 -> 4 天
- stage 3 -> 7 天
- stage 4 -> 15 天
- stage 5 -> 30 天
- stage 6 -> 90 天
- stage 7 -> 180 天

### 2.2 当前核心行为

当前调度逻辑的关键点是：

1. 先根据评分得到下一阶段
2. 再使用 `reviewedAt` 所在本地日期推进 `intervalDays`，并取目标自然日 `00:00` 作为新的 `dueAt`
3. 不根据 `dueAt` 与 `reviewedAt` 的差值做额外修正

这意味着：

- 过期本身不会改变阶段
- 只要用户给出 `GOOD` 或 `EASY`，卡片仍可能继续升级
- 系统默认认为“这次评分足以代表当前掌握状态”

---

## 3. 当前规则带来的问题

### 3.1 不是技术故障，而是产品语义失真

当前算法并不存在“不可恢复状态”。

即使高阶段卡片长期未复习，只要用户连续几次答错，也会重新回到低阶段。因此从可恢复性上看，当前规则是安全的。

问题在于：**过期越久，阶段越可能高估真实记忆水平。**

### 3.2 典型失真场景

| 场景 | 当前状态 | 用户实际记忆 | 当前算法可能结果 | 风险 |
|------|----------|--------------|------------------|------|
| stage 4 卡片晚了 20 天才复习 | 原定 15 天复习 | 记忆已明显变弱 | `GOOD -> stage 5` | 阶段继续升高，失真扩大 |
| stage 6 卡片晚了 90 天才复习 | 原定 90 天复习 | 很可能接近遗忘 | `GOOD -> stage 7` | 卡片被视为“稳定掌握” |
| stage 7 卡片半年后才回来 | 原定 180 天复习 | 记忆可信度极低 | `GOOD -> stage 7` | 长期死卡片继续占高阶段 |

### 3.3 产品层面的真实风险

1. **调度失真**：高阶段卡片在长期过期后仍可能继续升级。
2. **复习效率变差**：用户花时间处理“表面成熟、实际生疏”的卡片。
3. **心理压力变大**：回归用户面对一批过期卡片时，系统没有给出合理缓冲。
4. **规则解释别扭**：用户容易感到“明明拖了很久，怎么还算我掌握得很好”。

---

## 4. 设计原则

### 4.1 不惩罚用户，修正卡片状态

系统不应该表达为“你偷懒了，所以处罚你”，而应该表达为：

> 这张卡已经偏离原计划较久，当前阶段不再完全可信，需要先回落一点，再根据这次表现决定去向。

### 4.2 过期处理应与原间隔相关

绝对过期天数并不总是合理。

例如：

- 对 `stage 0` 而言，过期 7 天已经是严重偏离
- 对 `stage 7` 而言，过期 7 天几乎可以忽略

因此，过期判断应基于“**过期时长占当前计划间隔的比例**”，而不是只看绝对天数。

### 4.3 继续保持规则可解释

忆刻当前调度规则的优势是简单、直观、容易理解。

新的过期处理机制也应尽量保持：

- 不引入额外隐藏参数
- 不引入复杂状态机
- 用户能够理解为什么阶段会回落

### 4.4 保持可恢复性

任何过期衰减都必须是暂时的、可恢复的。

只要用户重新开始稳定复习，卡片应能自然回到高阶段，而不是被永久贴上“差卡片”标签。

## 5. 方案比较

### 方案 A：按绝对过期天数强制重置

示例：

```kotlin
val overdueDays = max(0, reviewedAtEpochMillis - dueAtEpochMillis).inWholeDays

val nextInterval = when {
    overdueDays >= 180 -> 1
    overdueDays >= 90 -> 2
    overdueDays >= 30 -> 4
    overdueDays >= 14 -> 7
    else -> originalInterval
}
```

优点：

- 改动最小
- 规则上手快

缺点：

- 只看绝对天数，忽略当前卡片原本的阶段差异
- 同样是过期 30 天，对短间隔卡和长间隔卡的语义不同
- 容易出现“阈值跳变”过硬的问题

结论：

- 可以作为应急方案
- 不建议作为长期默认方案

---

### 方案 B：引入 Ease 因子

示例思路：

```kotlin
data class Question(
    val easeFactor: Float = 1.0f
)
```

优点：

- 理论上更细腻
- 更接近传统 SRS 模型

缺点：

- 需要新增字段、迁移、展示与同步口径
- 算法透明度下降
- 会让当前“阶段制”变成“阶段 + 难度因子”的双轴系统

结论：

- 适合后续高级调度版本
- 不适合作为当前阶段的第一优先实现

---

### 方案 C：引入重新学习队列

示例思路：

```kotlin
enum class QuestionState {
    REVIEW,
    RELEARNING
}
```

优点：

- 记忆修复效果最好
- 更贴近 Anki / SuperMemo 的重学思路

缺点：

- 与当前“当天只出现一次”的产品原则冲突
- 需要引入新的状态机和短间隔队列
- 会显著提高实现和解释成本

结论：

- 适合未来做 v2 调度系统
- 不建议在当前版本直接引入

---

### 方案 D：按过期比例做阶段衰减

核心思路：

1. 先根据当前阶段得到其原计划间隔
2. 计算 `overdueRatio = overdueDays / plannedIntervalDays`
3. 先根据 `overdueRatio` 对当前阶段做一次“可信度回落”
4. 再套用原有评分规则得到最终阶段

优点：

- 兼顾公平性与可解释性
- 不需要新增持久化字段
- 与现有阶段制兼容
- 比“绝对天数惩罚”更符合记忆语义

缺点：

- 比纯阈值重置稍复杂
- 需要补充文档与测试，避免边界误解

结论：

- **推荐作为默认方案**

---

### 方案 E：仅做 UI 提示

思路：

- 显示“已过期 X 天”
- 标记“建议重新巩固”
- 提供只处理过期卡片的入口

优点：

- 几乎零算法成本
- 能缓解回归用户的心理压力

缺点：

- 不解决调度失真
- 本质上仍把判断成本留给用户自己

结论：

- 适合作为方案 D 的配套，不适合作为唯一方案

---

## 6. 推荐决策

### 6.1 核心决策

忆刻默认不采用“过期惩罚”，而采用：

**过期比例驱动的阶段衰减机制。**

换句话说：

- 不是用户被处罚
- 而是卡片阶段因为长期过期而失去可信度
- 系统先让卡片适度回落，再根据本次评分继续推进

### 6.2 为什么选择这个方案

因为它同时满足四个目标：

1. **符合直觉**：拖得越久，卡片越不该保持原阶段。
2. **足够公平**：不同阶段按“偏离比例”处理，而不是按同一绝对天数处理。
3. **实现成本低**：不需要引入新字段，也不需要重构为双状态机。
4. **延续现有理念**：仍然是简单、透明、可解释的阶段制。

---

## 7. 详细规则设计

### 7.1 术语定义

- `plannedIntervalDays`：当前阶段对应的计划间隔天数
- `overdueDays`：`reviewedAt` 本地日期与 `dueAt` 本地日期的自然日差值
- `overdueRatio`：`overdueDays / plannedIntervalDays`
- `effectiveStage`：在应用评分前，经过过期衰减后的临时阶段

### 7.2 阶段衰减规则

建议先使用如下默认阈值：

| overdueRatio | 含义 | 衰减结果 |
|--------------|------|----------|
| `< 1.0` | 虽然过期，但尚未超过一个完整计划间隔 | 不衰减 |
| `>= 1.0 && < 2.0` | 已偏离一个周期 | 降 1 级 |
| `>= 2.0 && < 4.0` | 已偏离多个周期 | 降 2 级 |
| `>= 4.0` | 当前阶段基本失去参考价值 | 直接降到 stage 0 或 1 |

说明：

- `stage 0` 本身已经是最低阶段，不再额外下降
- 对极端长期过期的高阶段卡片，可直接降到 `stage 0`
- 若希望保留一点“曾经学过”的痕迹，也可落到 `stage 1`

第一版建议使用更明确的规则：

```kotlin
fun decayStageByOverdue(
    currentStage: Int,
    overdueRatio: Double
): Int = when {
    overdueRatio < 1.0 -> currentStage
    overdueRatio < 2.0 -> (currentStage - 1).coerceAtLeast(0)
    overdueRatio < 4.0 -> (currentStage - 2).coerceAtLeast(0)
    else -> if (currentStage >= 3) 0 else currentStage.coerceAtMost(1)
}
```

这里对 `>= 4.0` 做特殊处理，是为了避免高阶段卡片在长期过期后仍只做轻微回退。

### 7.3 与评分规则的关系

衰减不是最终结果，而是评分前的修正。

完整流程为：

1. 读取当前 `stage`
2. 计算当前 `plannedIntervalDays`
3. 计算 `overdueDays` 与 `overdueRatio`
4. 得到 `effectiveStage`
5. 再基于 `effectiveStage` 套用原有评分规则
6. 用最终阶段对应的间隔计算新的 `dueAt`

即：

```text
currentStage
    -> overdue decay
effectiveStage
    -> rating transition
finalStage
    -> interval lookup
nextDueAt
```

### 7.4 评分后的最终规则

在 `effectiveStage` 基础上继续沿用原规则：

- `AGAIN` -> `0`
- `HARD` -> `max(effectiveStage - 1, 0)`
- `GOOD` -> `min(effectiveStage + 1, maxStage)`
- `EASY` -> `min(effectiveStage + 2, maxStage)`

这样可以保持用户对评分按钮含义的既有认知，不需要重新学习整套交互。

## 8. 规则示例

### 示例 1：轻度过期，不做额外打击

- 当前 `stage 3`
- 计划间隔 `7 天`
- 实际过期 `3 天`
- `overdueRatio = 3 / 7 = 0.43`
- 不衰减，`effectiveStage = 3`
- 若评分 `GOOD`，最终进入 `stage 4`

解释：

- 只是轻度拖延，不应被系统明显打回

### 示例 2：中度过期，先回落一级

- 当前 `stage 4`
- 计划间隔 `15 天`
- 实际过期 `20 天`
- `overdueRatio = 20 / 15 = 1.33`
- 先降 1 级，`effectiveStage = 3`
- 若评分 `GOOD`，最终回到 `stage 4`

解释：

- 这次 `GOOD` 更像“证明还保得住”，而不是“继续升级”

### 示例 3：重度过期，先回落两级

- 当前 `stage 6`
- 计划间隔 `90 天`
- 实际过期 `200 天`
- `overdueRatio = 200 / 90 = 2.22`
- 先降 2 级，`effectiveStage = 4`
- 若评分 `GOOD`，最终进入 `stage 5`

解释：

- 用户表现尚可，但系统不再默认它仍然处于高稳定区

### 示例 4：极端过期，视为需要重新巩固

- 当前 `stage 7`
- 计划间隔 `180 天`
- 实际过期 `800 天`
- `overdueRatio = 800 / 180 = 4.44`
- 先回落到 `stage 0`
- 若评分 `GOOD`，最终进入 `stage 1`

解释：

- 这种情况下继续保留高阶段已没有现实意义

## 9. UI 与文案建议

算法调整后，UI 需要配合做解释，否则用户会只看到“我明明答对了，为什么没升那么多”。

建议同时提供以下轻量提示：

1. **过期标签**
   - 例如：`已过期 20 天`
2. **状态提示**
   - 例如：`间隔偏离较久，本次会先按重新巩固处理`
3. **结果反馈**
   - 例如：`因长期过期，阶段先从 4 调整到 3，本次 GOOD 后恢复到 4`

原则是：

- 告诉用户系统做了什么
- 更重要的是告诉用户“为什么这么做”

## 10. 实施建议

### 10.1 第一阶段

优先落地：

- 过期比例衰减
- 基础 UI 提示
- 单元测试覆盖关键边界

这是当前版本性价比最高的组合。

### 10.2 暂不实现

当前不建议立即引入：

- Ease 因子
- 重新学习队列
- 按卡组完全自定义衰减规则

原因是这些能力都会显著提高调度复杂度，应在观察第一阶段数据后再决定是否继续演进。

### 10.3 后续演进方向

若第一阶段验证有效，可继续考虑：

1. 按卡组配置衰减强度
2. 为长期过期卡片提供“重新巩固模式”
3. 在统计页展示“阶段衰减触发次数”
4. 根据用户真实答题结果微调衰减阈值

---

## 11. 结论

忆刻在“长时间未复习”场景下，真正需要处理的不是“如何惩罚用户”，而是“如何修正已经失真的卡片阶段”。

因此，本设计推荐采用：

**基于过期比例的阶段衰减机制 + 轻量解释型 UI 提示**

不直接引入 Ease 因子，也不直接引入 relearning 队列。

这套方案的核心价值在于：

- 它比绝对天数惩罚更公平
- 它比复杂 SRS 参数更易解释
- 它能与当前阶段制低成本兼容
- 它把产品语义从“处罚”转回“帮助用户恢复记忆”
